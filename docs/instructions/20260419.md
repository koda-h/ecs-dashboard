## 2. `${POSTGRES_HOST_PORT}` 未展開バグの修正

### 問題
`dotenv` は `${VAR}` 形式の変数展開を行わない。`playwright.config.ts` が `dotenv.config()` で `DATABASE_URL` を未展開のまま `process.env` に書き込み、Next.js dev サーバーがこの値を継承して Prisma が `localhost:5432`（デフォルトポート）へ接続しようとしていた。

### 修正
`dotenv-expand` を導入し、`${POSTGRES_HOST_PORT}` を正しく展開するよう変更。

```ts
// playwright.config.ts
import { expand } from "dotenv-expand";
expand(dotenv.config({ path: ".env" }));
expand(dotenv.config({ path: ".env.local", override: true }));
```

### 注意点
環境変数を変更した後に `npm run test:e2e` を実行する場合、既存の dev サーバーが古い環境変数を持っているため、先にサーバーを停止してから実行すること。

## 3. パスキー認証 導入計画（改訂版）

> レビュー指摘（2026-04-19）を反映。

### 採用ライブラリ
`@simplewebauthn/server` + `@simplewebauthn/browser`

---

### ログイン UX の型：identifier-first

現行ログイン画面（`app/login/page.tsx`）はすでに「ユーザID / メールアドレス」入力欄を持つ。
これを活かし **identifier-first** 方式を採用する。

```
ユーザ操作:
  1. loginId/email を入力欄に入力
  2. 「パスキーでログイン」ボタンをクリック
     → 入力値を POST /api/auth/passkey/auth/options へ送信
     → サーバーが allowCredentials（そのユーザのcredential ID一覧）を返す
     → startAuthentication() を呼び出し、認証器がサインする
  3. レスポンスを POST /api/auth/passkey/auth/verify へ送信
     → 検証成功でセッション発行 → / へリダイレクト
```

**usernameless（discoverable credential）は今回スコープ外とする。**
`allowCredentials: []` と匿名チャレンジ管理は将来の拡張として別途設計する。

---

### フロー詳細

```
【パスキー登録】（ログイン済みユーザが自分のアカウントページで行う）
  ① POST /api/auth/passkey/register/options
       セッションから user.id を取得し、webAuthnUserId + challenge を生成して DB 保存
  ② ブラウザ: startRegistration() → navigator.credentials.create()
  ③ POST /api/auth/passkey/register/verify
       challenge 検証 → 公開鍵・counter 等を Passkey テーブルへ保存

【パスキー認証】（未ログインユーザがログイン画面で行う）
  ① POST /api/auth/passkey/auth/options  { loginId: string }
       loginId/email でユーザ検索 → allowCredentials を生成 → challenge を DB 保存
       ※ユーザが存在しない・パスキー未登録でも同一形式のエラーを返す（列挙防止）
  ② ブラウザ: startAuthentication() → navigator.credentials.get()
  ③ POST /api/auth/passkey/auth/verify
       challenge 検証 → 署名検証 → counter 更新
       → Passkey.userId（FK to User.id）で User を取得
       → setSessionCookie(user.userId, user.role) で既存セッション機構と接続
       → / へリダイレクト指示
```

### DB スキーマ

#### User モデルへの追加

WebAuthn user handle（PII を含まないランダム値）を User に持たせる。
`user.userId`（表示用）や `user.email` をそのまま userID に使わない。

```prisma
model User {
  // ... 既存フィールド ...
  webAuthnUserId String?  @unique // base64url 形式のランダム値。登録時に初回生成。
  passkeys       Passkey[]
}
```

#### 新規モデル

```prisma
// 1ユーザあたりの登録上限: 5件
model Passkey {
  id           String    @id @default(cuid())
  userId       String               // User.id（内部FK）
  name         String               // ユーザが設定する識別名（例: "iPhone Face ID"）
  credentialId String    @unique    // base64url
  publicKey    Bytes                // CBOR エンコード済み公開鍵
  counter      BigInt    @default(0) // リプレイ攻撃対策。verify 時に更新。
  deviceType   String               // "singleDevice" | "multiDevice"
  backedUp     Boolean   @default(false)
  transports   String[]             // ["internal", "usb", "ble", ...]
  createdAt    DateTime  @default(now())
  lastUsedAt   DateTime?

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
}

// チャレンジ一時保存。有効期限5分、使用後即削除。
model WebAuthnChallenge {
  id        String   @id @default(cuid())
  userId    String?  // 登録時 = User.id。認証時 = null（verify で credentialId から解決）
  challenge String   @unique
  type      String   // "registration" | "authentication"
  expiresAt DateTime
  createdAt DateTime @default(now())
}
```

### API Route（4本）

| Route | 役割 | 認証要否 |
|-------|------|---------|
| `POST /api/auth/passkey/register/options` | 登録チャレンジ生成 | **要**（セッションから user.id を取得） |
| `POST /api/auth/passkey/register/verify` | 公開鍵検証・DB 保存 | **要** |
| `POST /api/auth/passkey/auth/options` | 認証チャレンジ生成 | 不要 |
| `POST /api/auth/passkey/auth/verify` | 署名検証・セッション発行 | 不要 |

#### セッション発行の接続仕様（verify 内）

```ts
// credentialId から Passkey を引く
const passkey = await prisma.passkey.findUnique({ where: { credentialId }, include: { user: true } });
// User.userId（表示用ID）と role でセッションを発行する（既存実装と同じ形式）
await setSessionCookie(passkey.user.userId, passkey.user.role);
```

`Passkey.userId`（内部FK）を `setSessionCookie` に直接渡さない。

### 環境変数

既存の `NEXT_PUBLIC_APP_URL` を `expectedOrigin` として流用する（追加不要）。
`RP_ID` のみ新規追加。

| 変数 | 例 | 用途 |
|------|-----|------|
| `RP_ID` | `localhost` / `example.com` | WebAuthn の RP ID |
| `NEXT_PUBLIC_APP_URL` | `http://localhost:3000` | expectedOrigin（既存） |

verify 呼び出しイメージ:

```ts
await verifyRegistrationResponse({
  response,
  expectedChallenge,
  expectedOrigin: process.env.NEXT_PUBLIC_APP_URL!,
  expectedRPID: process.env.RP_ID!,
});
```

### セキュリティ仕様

| 項目 | 方針 |
|------|------|
| チャレンジ有効期限 | 5分。使用後即削除。期限切れは verify で拒否。 |
| counter 検証 | `@simplewebauthn/server` が自動検証。成功後に DB 更新。 |
| ユーザ列挙防止 | auth/options で loginId が存在しない・パスキー未登録のいずれも `"パスキーが見つかりません"` の同一エラーを返す。 |
| AccountLock | **パスキー認証は対象外**。ロック判定はパスワード試行に紐づいており、WebAuthn の失敗管理は認証器側の責務（指紋失敗回数等）。 |
| 既存認証との関係 | 並存。パスワードログインを廃止しない。 |
| webAuthnUserId | `crypto.randomUUID()` を base64url 変換して生成。登録時に User へ初回セット（以降は再利用）。 |
