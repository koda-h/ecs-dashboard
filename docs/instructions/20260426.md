## 1. Route Handler の実装

### 実装ファイル

| ファイル | メソッド | 認証 | 概要 |
|---|---|---|---|
| `app/api/passkey/register/options/route.ts` | POST | 必須 | 登録オプション生成。パスキー上限（5件）超過時は 409 |
| `app/api/passkey/register/verify/route.ts` | POST | 必須 | 登録検証・保存。clientDataJSON から challenge を取得し `consumeChallenge` で消費 |
| `app/api/passkey/authenticate/options/route.ts` | POST | 不要 | 認証オプション生成。discoverable credentials 方式（loginId 不要） |
| `app/api/passkey/authenticate/verify/route.ts` | POST | 不要 | 認証検証・セッション発行。credentialId → ユーザ特定 → `setSessionCookie` |
| `app/api/passkey/[id]/route.ts` | DELETE | 必須 | パスキー削除。所有権不一致は 403 |

### 実装のポイント

- チャレンジの紐付け: ブラウザから返される `credential.response.clientDataJSON` を base64url デコードして `challenge` を取り出し、DB の `WebAuthnChallenge` を特定する
- `consumeChallenge` が type・userId・有効期限を検証し、検証と削除をアトミックに行う
- `verifyRegistrationResponse`（`@simplewebauthn/server`）に渡す形式: `{ id, publicKey: new Uint8Array(passkey.publicKey), counter: Number(passkey.counter), transports }`

## 4. discoverable credentials への変更（identifier-first からの移行）

パスキーログイン時に loginId の入力が不要な discoverable credentials 方式に変更した。

### 変更の背景

identifier-first 方式ではユーザが loginId を入力すると、サーバが `allowCredentials` を構築して返す。これに対して discoverable credentials 方式では `allowCredentials` を渡さず、OS のパスキーピッカーが起動する。ユーザの特定は verify 後に credentialId から行う。

### 変更ファイル

#### `lib/auth/passkey/authentication.ts`

`generatePasskeyAuthenticationOptions` の引数 `loginId: string` を削除し、ユーザ検索と `allowCredentials` 構築を除去。チャレンジ保存は残す（`userId: null`）。

#### `app/api/passkey/authenticate/options/route.ts`

```ts
// 変更前: loginId を body から取得して検証
// 変更後:
export async function POST() {
  try {
    const options = await generatePasskeyAuthenticationOptions();
    return NextResponse.json(options);
  } catch {
    return NextResponse.json(
      { error: "認証オプションの生成に失敗しました" },
      { status: 500 }
    );
  }
}
```

#### `app/login/page.tsx`

`handlePasskeyLogin` から loginId の事前チェックと `{ loginId: ... }` の fetch body を削除。`loginIdValue` state はパスワードログインフォームの制御に引き続き使用。

#### `__tests__/auth/passkey/authentication.test.ts`

identifier-first を前提とした describe ブロック（`allowCredentials の構成`・`ユーザ列挙防止`・`AccountLock との独立性`）を削除し、discoverable credentials の仕様を検証するテストに書き直した。

| describe | テスト数 |
|---|---|
| generatePasskeyAuthenticationOptions > 正常系 | 4 |
| findUserByCredentialId > 正常系 | 4 |
| findUserByCredentialId > 異常系 | 1 |
| updatePasskeyAfterAuth > 正常系 | 3 |
| updatePasskeyAfterAuth > 異常系 | 1 |
| **計** | **13** |

#### `e2e/passkey/authentication.spec.ts`

- `fill(adminId)` を削除（loginId の入力なしで「パスキーでログイン」をクリック）
- loginId を前提としたエラーケーステスト2件を削除
- テスト数: 4件 → 2件（メインフロー + ボタン表示確認）

## 5. SSO セッションエラー検出の共通化

### 背景

`app/api/clusters/route.ts` と `app/api/services/route.ts` の両方に同一のエラー判定条件が重複していた。また、新たなエラーメッセージパターン（`"has expired"`）が追加されたことで冗長さが顕在化した。

### 変更内容

#### `lib/ecs.ts` に `isSsoSessionError` を追加

```ts
const SSO_EXPIRED_PATTERNS = [
  "SSO session associated with this profile is invalid",
  "SSO session associated with this profile has expired",
  "Could not load credentials from any providers",
];

export function isSsoSessionError(err: unknown): boolean {
  return (
    err instanceof Error &&
    SSO_EXPIRED_PATTERNS.some((p) => err.message.includes(p))
  );
}
```

両ルートの `if (err instanceof Error && ...)` ブロックを `if (isSsoSessionError(err))` に置き換え。
新しいパターンが増えた場合は `SSO_EXPIRED_PATTERNS` 配列に追記するだけで対応できる。
